# fi-observability (AURITY)

Read-only, on-demand operational inspection tools for AURITY production environments.

Design constraints:
- **Read-only by default**: never mutates the observed system.
- **No PHI/PII**: outputs are automatically redacted by key name heuristics.
- **On-demand only**: no resident agents, no real-time pipelines.
- **Deterministic analysis**: stable sorting + verifiable artifacts (`.sha256`).

Traceability:
- Aurity-Prompt-ID: `AUR-PROMPT-OBS-1.0`
- Date: `2025-12-18`
- Version: `1.0`

## Inputs (required parameters)

All tools share a homogeneous CLI surface via common flags:
- `--base-path` (absolute): repo root or backend folder that contains `fi_devtools`
  (validation only).
- `--log-source` (optional): JSONL structlog-compatible log source (`-` for stdin).
- `--slo-policy` (required): SLO policy file (JSON, or YAML if PyYAML is installed).
- `--environment` (required): `local | staging | production`.
- `--out-dir` (optional): output directory for artifacts (default: `artifacts/fi_observability`).

## Tools (exactly four)

### 1) Metrics Analyzer
Computes latency percentiles (p50/p95/p99) per endpoint and detects SLO violations.

Module: `backend.src.fi_observability.metrics_analyzer`

Example:
```bash
python -m backend.src.fi_observability.metrics_analyzer \
  --base-path "$PWD" \
  --log-source data/logs/structured.jsonl \
  --slo-policy config/fi.policy.yaml \
  --environment production \
  --out-dir artifacts/fi_observability
```
Artifacts:
- `metrics_report.json` (+ `metrics_report.json.sha256`)
- `metrics_report.html` (+ `metrics_report.html.sha256`)

### 2) Trace Inspector
Correlates events by `idempotency_key`, `workflow_id`, or `session_id` and represents
flow between `PUBLIC → INTERNAL → WORKER`.

Module: `backend.src.fi_observability.trace_inspector`

Example:
```bash
python -m backend.src.fi_observability.trace_inspector \
  --base-path "$PWD" \
  --log-source data/logs/structured.jsonl \
  --slo-policy config/fi.policy.yaml \
  --environment production \
  --since 2025-12-18T00:00:00Z \
  --until 2025-12-19T00:00:00Z \
  --out-dir artifacts/fi_observability
```
Artifacts:
- `traces_report.json` (+ `traces_report.json.sha256`)
- `traces_top.dot` (+ `traces_top.dot.sha256`) — render with Graphviz
  externally if desired.

### 3) Health Validator
Runs shallow/deep **GET** health checks and a finite load simulation (bounded
requests + concurrency). Evaluates latency/error rate against SLOs.

Module: `backend.src.fi_observability.health_validator`

Example:
```bash
python -m backend.src.fi_observability.health_validator \
  --base-path "$PWD" \
  --slo-policy config/fi.policy.yaml \
  --environment production \
  --base-url http://localhost:7001 \
  --shallow-paths /health \
  --deep-paths /health \
  --requests 30 \
  --concurrency 5 \
  --out-dir artifacts/fi_observability
```
Artifacts:
- `health_report.json` (+ `health_report.json.sha256`)

### 4) Alerts Simulator
Generates deterministic synthetic failure scenarios and validates the *logical
trigger* of alerts (no notifications).

Module: `backend.src.fi_observability.alerts_simulator`

Example:
```bash
python -m backend.src.fi_observability.alerts_simulator \
  --base-path "$PWD" \
  --slo-policy config/fi.policy.yaml \
  --environment production \
  --dry-run \
  --service public_api \
  --out-dir artifacts/fi_observability
```
Artifacts:
- `alerts_simulation.json` (+ `alerts_simulation.json.sha256`)

## Privacy & Safety

- These tools **never print raw request/response bodies**.
- Outputs are redacted by key name heuristics (e.g., `email`, `telefono`, `curp`,
  `token`, `transcript`).
- Still treat input logs as sensitive and store artifacts in controlled locations.

## Limits

- No cloud assumptions.
- No external monitoring integrations.
- No persistent metrics beyond generated artifacts.
- Intended for **analysis on demand**, not continuous monitoring.

## Architecture: Runtime vs Analysis

AURITY has **two complementary observability systems**:

### 1. Runtime Logging (`backend/observability/`)
**Purpose:** Generate structured logs during production requests.

Components:
- `logging.py` - Configures structlog with PHI redaction
- `chat_events.py` - Event logging functions (log_chat_request, log_chat_response)
- `log_spec.py` - Schema contract for required fields

Output: JSONL logs to `/tmp/backend.log` or stdout

### 2. Analysis Tools (`backend/src/fi_observability/` - THIS PACKAGE)
**Purpose:** Analyze logs generated by (1) on-demand.

Components:
- `metrics_analyzer.py` - Compute latency percentiles, detect SLO violations
- `trace_inspector.py` - Correlate distributed traces by trace_id
- `health_validator.py` - Validate health endpoints with load simulation
- `alerts_simulator.py` - Test alert triggers without sending notifications

Input: JSONL logs from (1)
Output: JSON/HTML reports with SHA256 checksums

### Data Flow
```
Production Request
      ↓
backend/observability/logging.py (structlog)
      ↓
JSONL log entry → /tmp/backend.log
      ↓
[Time passes - logs accumulate]
      ↓
fi_observability tools (on-demand analysis)
      ↓
Reports (metrics_report.json, traces_report.json, etc.)
```

**Key Principle:** Runtime system generates data, analysis tools consume it.
No tight coupling - analysis can run hours/days after events occur.
